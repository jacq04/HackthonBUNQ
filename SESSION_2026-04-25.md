## Pod creation + lifecycle

### Schema

`0006_circle_lifecycle.sql` (state machine + cycles + bids + mandates),
`0008_admin_created_circles.sql` (`min_trust_score`, `theme`,
`cultural_hint`, `max_members`, `payout_strategy`, `description`).

### Authority

**Admin-only.** `form_new_circle` removed from the Matchmaker tool list;
the dispatcher returns `{ok:false}` if anything calls the legacy tool.
System prompt states "CIRCLE CREATION IS ADMIN-ONLY".

### Bootstrap demo pods

`backend/scripts/bootstrap_demo_circles.py` seeds five recruiting pods
(Lagos Crew, Berlin Builders, Amsterdam Anchors, Mumbai Makers, Lisbon

### Matchmaker

`list_open_circles` enforces **hard filters**: cycle_count exact match,
contribution within Â±15%, trust â‰Ą min_trust_score, capacity remaining.
Rejected pods returned in `rejected[]` with reason. `_add_member` enforces
the same filters as a backstop. Two actions only: **JOIN** an eligible
pod or **WAITLIST**.

### Batch fill

New tool `propose_fill_circle(group_id, additional_user_ids, rationale)`.
When `seats_open == 1 + N compatible waitlisters`, the agent calls this
to insert all of them as `members.status='invited'`, mark
`users.waitlist_status='matched'`, set the pod to `awaiting_accepts`
with a 48h `accept_deadline`, emit `matchmaker.circle_filled` and per-user
`matchmaker.invited` events, audit `circle.fill`.

### Status-aware `_add_member`

Recruiting pods land users as `invited` (no `payout_cycle`); active pods
keep the previous "next open slot" path.

### Accept flow

`POST /groups/{id}/invites/respond` writes a `mandates` row (with bunq
`create_autoflow` stub if available), flips the member to `accepted`,
fires the **first debit** as a TigerBeetle linked batch (gateway â†’ pool +
gateway â†’ member_contrib). Now also:
- Inserts a `contributions` row with `status='posted'` so the admin
  money-flow rollup is correct.
- Calls `bunq.make_payment` from the accepter's account â†’ platform
  admin's IBAN, then emits `bunq.payment.posted`. Verified Asha's
  balance increments per accept.
- On the threshold (`accepted >= cycle_count`),
  `_transition_to_chartered` retires excess `invited` members to
  `exited_clean` AND flips their `users.waitlist_status='waiting'` so
  the Matchmaker re-considers them on the next round. Emits
  `invite.retired` with the user list.

### Wallet view of pods

`GET /groups` now filters `members.status` to
`accepted|active|received|exited_clean`. Invited-but-not-yet-accepted
pods don't show in the wallet's pods list. New `GET /me/invitations`
returns invited pods; the wallet renders an `_InvitationsBanner` (deep
bowl card, one row per invite with `review` CTA) that pushes to the
existing accept page.
